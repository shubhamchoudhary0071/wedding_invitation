Algorithms starter pack

Covers the ‚Äústarting with algorithms‚Äù section: common algorithm families + when to reach for them.

Sorting

  • Bubble / Selection / Insertion (simple, small inputs, teaching)
  • Merge sort (stable, guaranteed O(n log n))
  • Quick sort (fast average, watch worst‚Äëcase without randomization)
  • Heap sort (O(n log n), in‚Äëplace-ish, not stable)

Searching

  • Linear search (unsorted or tiny inputs)
  • Binary search (sorted / monotonic ‚Äúanswer space‚Äù)
  • Binary search on answer (minimum feasible / maximum feasible patterns)

Graph algorithms (named in the post)

  • BFS / DFS (traversal, reachability, layers)
  • Dijkstra (shortest path, non‚Äënegative weights)
  • Bellman‚ÄëFord (handles negative edges, detects negative cycles)
  • Prim / Kruskal (minimum spanning tree)
  • Extra concepts also mentioned: SCC, max‚Äëflow

How to choose an approach (from the post)

  • Read statement carefully; constraints often hint the tool.
  • Look for patterns in input (sorted? graph? grid?).
  • Practice under time pressure; learn common templates.

Pros

Cons

  • Extra memory for pointers; slower cache locality

DP on grids

Grid‚ÄëDP patterns from the post: count paths, minimum path sum, obstacles, maximal square.

Core patterns
  • Count paths (right/down moves): dp[i][j] = dp[i-1][j] + dp[i][j-1]
  • Min path sum: add current cell to min(from top, from left)
  • Obstacles: if blocked cell ‚Üí 0 ways; otherwise accumulate
  • Largest square of 1s: dp = 1 + min(top, left, diag)

BFS (unweighted shortest path)

queue = [start]
dist[start] = 0
while queue:
  u = pop_left()
  for v in neighbors(u):
    if v not visited:
      dist[v] = dist[u] + 1
      push_right(v)

Dijkstra (non-negative weights)

pq = [(0, start)]
dist[start] = 0
while pq:
  d,u = pop_min()
  if d != dist[u]: continue
  for (v,w) in edges[u]:
    if d + w < dist[v]:
      dist[v] = d + w
      push(pq, (dist[v], v))

Segment trees

The post includes a segment tree walkthrough (max query) + notes on cycle detection in graphs.

What to remember
  • Segment tree supports range query + point update in O(log n).
  • Common aggregates: sum/min/max/gcd; extend with lazy propagation for range updates.
  • Implementation idea: build recursively; query with 3 cases (outside / fully inside / partial overlap).
Graph cycle detection (also near this section in the post)
  • Undirected cycle: DFS with parent tracking, or BFS storing (node,parent).
  • DAG cycle: Kahn‚Äôs algorithm (topo). If processed nodes < V ‚Üí cycle exists.

Job Scheduler: study-friendly outline

Functional requirements

  • Schedule one-time and recurring (cron-like) jobs.
  • Jobs reference an executor class (e.g., an IJob interface implementation).
  • Store execution results and allow querying.

Key entities (as listed)

  • Job: id, name, executor class, priority, status, last start/end, last executor, parameters
  • Trigger: one-time / cron / interval; start/end boundaries
  • Executor: worker node id (e.g., IP-based), last heartbeat
  • Relationship: Job (1) ‚Üí (many) Triggers

High-level architecture

  • Producers publish schedule requests to a message bus (Kafka topic).
  • Scheduler service consumes requests, generates unique IDs (Snowflake-style), writes Job+Trigger to sharded RDBMS.
  • Dispatcher/runner pulls due triggers, assigns to workers, tracks heartbeats.
  • Result handler consumes execution results and stores them in a write-optimized store (e.g., Cassandra).
  • Coordination (ZooKeeper) stores cluster metadata (shards, partitions, workers).

Why message queues (points in post)

  • Decouple producers/consumers and scale independently.
  • Lower latency for producers; durability via persisted logs.
  • Ordering guarantees (within partitions) + backpressure control.
The post also calls out time/clock reliability in distributed systems (NTP drift, delays) ‚Äî essential for correct scheduling.
